# Requirements: OpenSRE

**Defined:** 2026-06-07
**Core Value:** Evidence-based incident reasoning engine that strictly enforces the Evidence → Correlation → Timeline → Graph → Hypothesis → Ranking → Explanation order.

## v1 Requirements

### Foundation & Core Orchestration (Sprint 1)

- [ ] **ROUT-01**: The system must provide a unified `LLMProvider` interface to run completions/structured responses.
- [ ] **ROUT-02**: The `LLMProvider` must implement specific adapters for OpenAI, Anthropic, Gemini, Ollama, and OpenRouter.
- [ ] **ROUT-03**: No other backend component is allowed to import or invoke an LLM SDK directly.
- [ ] **STOR-01**: The system must provide an immutable `EventStore` under `backend/app/events/` that writes raw evidence files.
- [ ] **STOR-02**: The system must never mutate or delete raw events once written.
- [ ] **FLAG-01**: The system must implement Feature Flags for optional components: memory, remediation, MCP, Slack, and PagerDuty.
- [ ] **AUDT-01**: The system must log every action (investigation creation, evidence collection, worker invocations, report generation, etc.) to an immutable, append-only `AuditTrail` DB.
- [ ] **PIPE-01**: The system must expose an `InvestigationPipeline` that executes the sequential stages: collect, normalize, correlate, timeline, graph, hypothesize, rank, and report.
- [ ] **PIPE-02**: Every active run must register in the `Investigation` registry database, tracking state transitions (QUEUED, COLLECTING, etc.).
- [ ] **SNAP-01**: The system must generate a reproducible `InvestigationSnapshot` containing raw events, timeline, graph, findings, report, and metadata.
- [ ] **SNAP-02**: Snapshots must store schema_version, pipeline_version, ranking_version, graph_version, prompt_version, model name, and temperature.
- [ ] **REPT-01**: The system must generate reports conforming to the `IncidentReportV1` schema, including root cause, confidence, factors, timeline, evidence, blast radius, remediation, similar incidents, and service owners.
- [ ] **WORK-01**: Investigation pipeline execution must run asynchronously, enqueued as tasks in a background queue (Celery/RQ + Redis).
- [ ] **WORK-02**: The API endpoint `POST /investigations` must return a `job_id` immediately without blocking.
- [ ] **DOCK-01**: The system must provide a Docker deployment configuration to run API, workers, Redis, and integrations locally.
- [ ] **DOCK-02**: The system must include CI/CD pipelines configured for GitHub Actions (`.github/workflows/`) or GitLab CI/CD (`.gitlab-ci.yml`) to run tests and linters automatically.

### Integrations (Sprint 2)

- [ ] **INTG-01**: The system must implement a Prometheus integration plugin to query and collect metric evidence.
- [ ] **INTG-02**: The system must implement a Loki integration plugin to query and collect log evidence.
- [ ] **INTG-03**: The system must implement a Kubernetes integration plugin to query cluster resource events and statuses.
- [ ] **INTG-04**: The system must implement a GitHub integration plugin to retrieve commit logs and repository histories.

### Timeline & Correlation (Sprint 3)

- [ ] **TIME-01**: The system must implement a `TimelineBuilder` that parses collected metrics, logs, traces, deployments, and alerts into a deterministic, chronologically ordered sequence.
- [ ] **CORR-01**: The system must correlate evidence events across different sources based on timestamps, service labels, and metadata.

### Graph & Root Cause Ranking (Sprint 4)

- [ ] **GRP-01**: The system must define a `GraphProvider` abstraction to decouple domain logic from specific graph database engines.
- [ ] **GRP-02**: The system must implement a `NetworkXProvider` that models services, deployments, databases, queues, alerts, traces, and infrastructure dependencies in memory.
- [ ] **RNK-01**: The system must implement a `RootCauseRanker` that computes hypothesis scores using four factors: temporal alignment, evidence strength, source reliability, and historical similarity.
- [ ] **RNK-02**: Every confidence score generated by the ranker must be explainable, detailing the specific calculation weights of the four factors.
- [ ] **TEST-01**: The system must implement automated evaluation tests using benchmark scenarios (e.g. redis_saturation, memory_leak, crashloopbackoff).
- [ ] **TEST-02**: The evaluation suite must track quality metrics: root_cause_accuracy, timeline_accuracy, evidence_coverage, confidence_calibration, and false_positive_rate.
- [ ] **CLI-01**: The system must implement a CLI command `opensre investigate <path>` to trigger local investigations.
- [ ] **CLI-02**: The system must implement a CLI command `opensre explain <incident-id>` generating persona-specific explanations (sre, developer, executive, support).
- [ ] **SDK-01**: The system must implement the plugin interfaces: `DataSourcePlugin`, `AgentPlugin`, `GraphPlugin`, and `RemediationPlugin`.

## v2 Requirements

### Advanced Features & Productionization

- **NOTF-01**: Integrate Slack notifications and interactive blocks.
- **NOTF-02**: Integrate PagerDuty alerting and acknowledgements.
- **NEO4-01**: Implement `Neo4jProvider` for persistent database graph storage.
- **MEM-01**: Implement memory-based learning from historical incidents to enhance RCA similarity ranking (`ENABLE_MEMORY`).
- **REMED-01**: Implement human-in-the-loop remediation orchestration (`ENABLE_REMEDIATION`).

## Out of Scope

| Feature | Reason |
|---------|--------|
| Direct LLM API Calls | Strict requirement: all LLM interactions must use `LLMProvider` abstraction. |
| Alert-to-LLM-to-Guess reasoning | Strict requirement: findings must be evidence-backed and follow the ordering invariant. |
| Sprint 1–4 Frontend UI | Frontend development is deferred until core engine and command line are complete and validated. |
| Automatic Auto-Remediation without approval | Security and stability risk. Only human-approved remediation workflows are considered. |
